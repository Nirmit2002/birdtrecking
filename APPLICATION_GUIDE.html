Overview

What exactly is this application?

The White Stork Migration Visualiser is a web application that tracks the real-life journeys of wild storks across the sky — showing where they flew, how fast, where they stopped, and what their GPS tags recorded along the way.

Think of it like Google Maps, but instead of tracking your own journey, it tracks the migration of white storks fitted with GPS tracking devices in Portugal in 2022. Scientists attach tiny GPS tags to birds. Every few hours, the tag records the bird's exact location, speed, altitude, and battery level — and sends it back via mobile network.

This application takes that raw data, processes it, stores it in a database, and presents it through a visual, interactive dashboard that anyone can explore through a web browser.

In plain terms: A bird has a GPS tracker. The tracker sends data. This app collects that data, draws the bird's path on a map, and shows you stats about its journey — all accessible from a browser on any device.

Technology

The tech stack — and why each piece was chosen

Six technologies work together to make this app run. Here's what each one is, in plain English, and the specific reason it was used here.

Python
Language

What it is

Python is a programming language — the set of instructions the computer reads. Think of it as the language this entire application is written in, like English is the language of this document.

Why we used it

Python has the best scientific data tools available — pandas for spreadsheet-like data, geopandas for geographic (map) data, scipy for statistics. For a project that processes GPS coordinates and calculates migration routes, Python was the natural home.

Flask
Web Framework

What it is

Flask is a tool that turns a Python program into a website. Without Flask, Python can't respond to web browsers. Flask listens for requests (someone opens a page) and sends back the right content.

Why we used it

Flask is lightweight and gives full control — perfect for a focused academic project. It handles user login, page routing (which URL shows which page), and the API endpoints that feed data to the map. It doesn't force a heavy structure like larger frameworks would.

MySQL
Database

What it is

MySQL is a database — a structured place to store information that can be searched, filtered, and retrieved fast. Think of it as a very organised filing cabinet that the application opens whenever it needs to look something up.

Why we used it

The app needs to store user accounts (with secure hashed passwords) and track uploaded files across sessions. MySQL is mature, reliable, and well-supported. It pairs naturally with Flask via SQLAlchemy, which lets Python talk to the database without writing raw SQL queries.

Pandas & GeoPandas
Data Processing

What it is

Pandas is like Excel inside Python — it reads spreadsheets, filters rows, calculates averages. GeoPandas is the same thing but for geographic data — it understands coordinates and can measure distances on the Earth's surface.

Why we used it

The raw stork data arrives as an Excel file. Pandas reads it, cleans it, and turns it into usable tables. GeoPandas then works out actual distances between GPS points on the curved surface of the Earth (not just straight-line subtraction).

GeoJSON
Map Format

What it is

GeoJSON is a file format specifically designed to store geographic shapes — points, lines, and areas — in a way that mapping tools can read. It's just a text file with coordinates formatted in a special way.

Why we used it

The migration tracks (the paths the birds flew) need to be drawn on a map. GeoJSON is the standard format for this — all modern map libraries like Leaflet and Mapbox understand it natively. The app converts processed GPS data into GeoJSON files so the frontend map can draw the routes.

HTML / CSS / JavaScript
Frontend

What it is

These three languages build what you see in the browser. HTML is the structure (the skeleton), CSS is the styling (colours, layout, fonts), and JavaScript is the behaviour (what happens when you click a button, how the map responds).

Why we used it

Every website uses these — there's no alternative for browser-based interfaces. The app's JavaScript (app.js, charts.js) fetches data from the Flask API and renders it on the map and in charts without reloading the page.

The Journey of Data

From GPS tag on a bird to dot on a map

Raw data goes through five transformation steps before you see a line drawn on the map. Here's each step explained simply.

1

GPS tag records a fix

Every few hours the tag on the stork's back locks onto satellites and records: latitude, longitude, altitude, speed, heading, battery level, and temperature. This happens automatically — the bird doesn't even notice.

2

Data arrives as an Excel file

The tag transmits data via mobile network to a research database (Movebank). Scientists export it as data.xlsx — a spreadsheet with 99 rows and 30 columns. This file lives in the data/ folder.

3

preprocess.py transforms it

When the app starts for the first time, preprocess.py reads the Excel file using pandas, groups fixes by bird, calculates distances and stopovers using GeoPandas, then writes out four output files: tracks.geojson, stopovers.geojson, stats.json, and timeline.json. These are saved and reused — the heavy calculation only runs once.

4

Flask serves the data via API

When the browser opens the map page, JavaScript calls API endpoints like /api/tracks and /api/stats. Flask reads the GeoJSON files and returns them as JSON responses — like a waiter bringing food from the kitchen to your table.

5

JavaScript draws it on the map

The browser's JavaScript receives the GeoJSON and uses a map library to draw the migration routes as lines on an interactive map, place markers at stopovers, and render charts for speed and altitude. The user can click, zoom, and explore.

Under the Hood

Key code explained in plain English

You don't need to be a programmer to follow these explanations. Each snippet is the actual code from the app, broken down line by line.

1. How a page route works (app.py)

A "route" is the link between a URL and a Python function. When someone visits /dashboard, Flask knows exactly which function to run.

@app.route('/dashboard')
@login_required
def dashboard():
    files = UploadedFile.query.filter_by(
        uploaded_by=current_user.id
    ).all()
    return render_template('dashboard.html', files=files)
Line by line: @app.route('/dashboard') — "when someone visits /dashboard, run this function." @login_required — "but only if they're logged in — otherwise redirect to login." The function fetches that user's uploaded files from the database, then sends the dashboard HTML page with those files included.

2. How the database model works (models.py)

A "model" is a Python class that describes a database table. Instead of writing SQL by hand, we describe the table in Python and SQLAlchemy creates it automatically.

class User(db.Model, UserMixin):
    id         = db.Column(db.Integer, primary_key=True)
    username   = db.Column(db.String(80), unique=True, nullable=False)
    email      = db.Column(db.String(120), unique=True)
    password   = db.Column(db.String(255), nullable=False)
    role       = db.Column(db.String(20), default='researcher')

    def set_password(self, raw_password):
        self.password = bcrypt.generate_password_hash(raw_password)

    def check_password(self, raw_password):
        return bcrypt.check_password_hash(self.password, raw_password)
Plain English: This creates a "users" table with columns for id, username, email, password, and role. Crucially, passwords are never stored as plain text — bcrypt scrambles them into a hash. Even if someone stole the database, they couldn't read the passwords. check_password verifies a login by re-scrambling the typed password and comparing the hashes.

3. How GPS data becomes map-ready (preprocess.py)

This is where raw spreadsheet data is transformed into the GeoJSON format that the map understands.

def run_preprocessing(xlsx_path):
    df = pd.read_excel(xlsx_path)          # Read the Excel file

    gdf = gpd.GeoDataFrame(df,             # Convert to geographic dataframe
        geometry=gpd.points_from_xy(
            df['location-long'],
            df['location-lat']
        )
    )

    for bird_id, group in gdf.groupby('individual-local-identifier'):
        track = group.sort_values('timestamp')
        line = LineString(track.geometry.tolist())
        tracks_features.append({
            "type": "Feature",
            "properties": {"bird_id": bird_id},
            "geometry": mapping(line)
        })
What's happening: We read the Excel file, then tell GeoPandas that the location-long and location-lat columns are GPS coordinates. We then group all rows by bird name, sort them by time (so the line goes in the right direction), connect all the points into a line, and save it as a GeoJSON "Feature" — exactly the format a map library expects.

4. How the API endpoint works (app.py)

The map page fetches data dynamically — JavaScript calls an API and gets back JSON. Here's that endpoint.

@app.route('/api/tracks')
@login_required
def api_tracks():
    data = _read_geojson('tracks.geojson')
    return jsonify(data)

def _read_geojson(filename):
    path = os.path.join(GEOJSON_DIR, filename)
    with open(path) as f:
        return json.load(f)
Plain English: When JavaScript visits /api/tracks, Flask opens the pre-built tracks.geojson file, reads it, and sends it back as a JSON response. The JavaScript then uses that response to draw the bird tracks on the map. This separation (Python serves data, JavaScript draws it) is called a "REST API" pattern.

Application Features

What you can do in the app

The application has five main areas, each with a specific purpose.

Migration Map

Interactive map showing all three birds' GPS tracks, stopover locations, and individual points. Click any marker for details.

Dashboard

Overview of all birds with stats cards: total distance, average speed, max altitude, stopover count, and tracking dates.

User Management

Admin panel to view all registered users, change their roles (admin or researcher), and remove accounts.

Data Upload

Upload new Excel GPS data files. The system preprocesses them automatically and updates the map and statistics.

Timeline & Charts

Time-series charts of speed and altitude for each bird over the tracking period, showing patterns in flight behaviour.

Authentication

Secure login and registration with bcrypt-hashed passwords. Two roles: admin (full access) and researcher (view only).

Setup Reference

How to run the application

Four commands in sequence. The first three are one-time setup; after that just run the last command each time.

# 1. Clone the code from GitHub
git clone https://github.com/Nirmit2002/birdtrecking.git
cd birdtrecking

# 2. Install all Python dependencies
pip install -r requirements.txt
pip install pymysql cryptography

# 3. Set up the database (Windows PowerShell)
$env:PATH += ";C:\Program Files\MySQL\MySQL Server 8.0\bin"
mysql -u root -p -e "CREATE DATABASE stork_db; CREATE USER 'stork_user'@'localhost' IDENTIFIED BY 'stork_pass2024'; GRANT ALL PRIVILEGES ON stork_db.* TO 'stork_user'@'localhost'; FLUSH PRIVILEGES;"
Get-Content stork_db.sql | mysql -u root -p stork_db

# 4. Start the app (run this every time)
python backend/app.py

Then open http://localhost:5000 in any browser.
